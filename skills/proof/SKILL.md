# 文档润色

你正在帮助用户润色指定的文档文件，确保其符合中文文档书写规范。

## 润色目标

**核心原则**：
- **内容准确性**：保持原意不变，修正错误信息
- **格式规范化**：统一 Markdown 格式标准
- **语言精简化**：去除冗余表达，提升可读性
- **排版标准化**：符合中英文排版规范

## 阶段一：加载文件

初始请求来自当前用户对话。

**操作**：
1. 解析用户请求中的目标文件路径；如果未提供，则使用当前上下文中正在处理的文件
2. 读取文件完整内容
3. 如果文件不存在或无法读取，向用户说明并停止

## 阶段二：润色执行

### 1. Markdown 格式检查与修正

**标题层级**：
- 检查是否跳级
- 确保不超过四级标题

**列表格式**：
- 无序列表使用 `-`
- 有序列表使用 `1.`
- 嵌套列表使用四空格缩进

**代码块**：
- 使用三反引号包裹
- 指定语言类型
- 内联代码使用单反引号

**表格**：
- 检查对齐格式
- 确保使用标准 Markdown 语法

### 2. 中英文排版修正

**空格规范**：
- 中英文之间添加半角空格
- 中文与数字之间添加半角空格

**标点符号**：
- 中文使用全角标点
- 英文使用半角标点

**字符规范**：
- 英文和数字使用半角字符
- 删除多余空格和全角空格

### 3. 语言风格优化

**精简表达**：
- 删除冗余的“的”“是”“了”“因为”等
- 删除“进行”“进行了一次”等空洞动词
- 合并重复表达

**句式优化**：
- 优先使用短句
- 避免过长复合句
- 使用主动语态

**专业术语**：
- 保持术语一致
- 首次出现时可加简要说明

### 4. 示例对比优化（如有）

检查文档中的示例代码：
- 语法是否正确
- 是否包含必要导入
- TypeScript 类型定义是否完整

## 阶段三：呈现修改

**操作**：
1. 将润色后的完整内容呈现给用户
2. 使用表格列出主要修改：

   ```markdown
   | 修改类型 | 原文 | 修改后 |
   |----------|------|--------|
   | 排版 | "因为采访,我" | "采访, 我" |
   | 精简 | "前段时间因为采访的缘故" | "前段时间采访" |
   | 格式 | `# 标题`（跳级） | `## 标题` |
   ```

3. 询问用户是否确认应用这些修改

## 阶段四：应用修改

**条件**：仅在用户确认后执行

**操作**：
1. 重新读取文件以获取最新内容
2. 根据确认后的修改更新文件
3. 向用户确认修改完成

- [ ] **[P0]** 列表格式：无序用 `-`，有序用 `1.`，嵌套缩进正确
- [ ] **[P0]** 代码块：三反引号包裹，指定语言
- [ ] **[P0]** 表格：对齐，标准语法
- [ ] **[P0]** 中英文空格：混排时已添加半角空格
- [ ] **[P0]** 标点符号：中文全角，英文半角
- [ ] **[P0]** 冗余词汇：删除多余表达
- [ ] **[P0]** 句式：短句为主，避免冗长
- [ ] **[P0]** 术语：保持一致
- [ ] **[P0]** 代码示例：语法正确，类型完整

## 注意事项

1. **保持原意**：润色不改变原文技术含义和逻辑
2. **保守修改**：对不确定的修改，优先询问用户
3. **版本控制**：建议在修改前提交 git，便于回滚
4. **大文件处理**：超过 2000 行的文件建议分段处理
