# 作者风格基线

## 稳定特征

- 开头先交代“为什么值得看”或“本文讲什么”，很少绕大圈。
- 抽象主题常见“背景引入 + 一句话说明/核心定义”的开场方式。
- 首段会推进到结论，但通常不会在同一段里反复换说法重复强调同一个判断。
- 抽象概念后面很快给定义、机制拆解或具体例子。
- 常用“关键”“核心”“本质上”“比如”“可以看到”“相当于”作为解释支点，但不会一段堆很多。
- 技术说明常见“判断 + 原理 + 示例/配置/图示”的推进方式。
- 愿意用“我们”“本文”“笔者”建立叙述关系，但不会刻意卖弄口语。
- 粗体主要用来标关键概念，不拿来放大情绪。
- 表格、列表、图示、代码块都服务于说明，不做装饰。
- 经常保留中文概念 + 英文术语括号，如“上下文工程（Context Engineering）”。
- 已有英文小节标题或固定术语时，通常直接保留，再在正文里解释。
- 常在开头或关键处用引用块说明来源、仓库或阅读建议。
- 很少写“总的来说”式收尾，更常见的是讲完关键点就进入下一节，或直接把结论落下。

## 推荐写法

- 第一段直接给背景和结论，不用空泛开场。
- 如果主题抽象，可补“一句话说明”帮助读者快速建立心智模型。
- 先讲本质，再讲机制，最后给例子、命令、配置或适用场景。
- 判断性语句后面补支撑：为什么、怎么发生、适用在哪、代价是什么。
- 对有通用英文名称的技术概念，可保留“中文名（英文名）”的写法。
- 对引用资料、源码仓库、实验项目，可用简短引用块集中交代，不要把说明塞进正文第一段。
- 允许适度保留作者习惯用语，只要不影响清晰度。

## 不像作者的写法

- “在当今快速发展的……”
- “这不仅仅是……更是……”
- “让我们深入探讨……”
- 大段情绪化拔高、行业口号、未来展望。
- 每段都用同一种句式起手。
- 为了显得生动，故意加俚语、网络腔或感叹号。
- 把图示说明、资料来源、实验仓库打散到正文里，导致主线变乱。

## 可优先保留的表达方式

以下表达在作者文章里出现频率较高，可以自然保留，但不要硬塞：

- “本文主要……”
- “关键在于……”
- “本质上……”
- “可以看到……”
- “比如……”
- “相当于……”
- “需要注意的是……”
